# news-collection（外交・世界情勢ニュースの調査・記事化）

## 概要
`news.basaapp.com` は **外交・安全保障・紛争／戦争・世界情勢（国際政治）** を扱うニュースサイト。日本の外交・安保も対象に含む。routine が毎日実行し、信頼できる情報源を調査して **1回あたり3本程度**（上限は設けないが質を優先）の記事を追加する。記事は Markdown で `content/articles/` に置き、main へ push すると自動デプロイされる。

**扱う主な領域**: 各国の外交・首脳/閣僚会談・条約/合意、安全保障・防衛、武力紛争/戦争の動向、国連・NATO・EU 等の国際機関、制裁・通商をめぐる国際対立、選挙や政変の国際的影響、日本の外交政策。
**扱わないもの**: 国内政治の内輪の話題（国会運営・政局・国内人事など、国際的含意の薄いもの）、芸能・スポーツ等。

## 編集方針（国際・紛争報道の必須ルール）
中立性と正確性が最重要。紛争報道は特に慎重に：

1. **事実と意見を分離**: 確認できた事実を中心に書く。分析・見通しは「解説」「背景」など明示した区画に限定し、断定しない。
2. **当事者の主張は帰属を明示**: 紛争・対立では各当事国/勢力の発表を一方的に事実として書かず、「〇〇政府は〜と発表した」「〜と主張している」と必ず主体を明示。プロパガンダ・情報戦の可能性を念頭に置く。
3. **死傷者・被害の数値は出典と留保を付す**: 戦闘の犠牲者数などは発表主体（政府・国連・人道機関など）を示し、独立検証の有無に触れる。確定値のように断定しない。
4. **中立・多角的視点**: 当事国双方／関係各国の立場や、国際社会の異なる反応を併記する。特定の国・勢力を擁護/非難する価値判断に立ち入らない。
5. **出典を必ず明記**: 各記事に最低2件の出典 URL。一次情報（政府・国際機関の公式発表）を優先し、国際通信社・主要報道で補う。
6. **憶測・未確認情報を出さない**: 裏取りできない速報、SNS の伝聞、扇情的見出しは扱わない。未確定は「〜と報じられている」「〜とされる」と帰属。
7. **出典URLは実在を必ず確認**: `sources` の URL は WebFetch 等で実際に解決し記事内容と一致することを確認してから書く。記憶や推測で URL を組み立てない。ドメインは正規のもの（例: NHK は `www3.nhk.or.jp`、ロイターは `www.reuters.com`）。
8. **固有名詞・地名の表記**: 人名・国名・地名・組織名は一般的な日本語表記に統一。初出で必要なら原語/英語を併記。
9. **中立的なタイトル**: 煽り・断定・レッテルを避け、何が起きたかを淡々と示す。

## ワークフロー（routine 実行時 / 記事1本ごと）

1. **トピック収集**: ホワイトリストの一次情報・国際通信社・WebSearch で、当日の主要な外交・安全保障・世界情勢トピックを洗い出す（候補を多めに）。
2. **重複チェック（article-registry スキルを使用）**: 各候補について
   ```bash
   bun run .claude/skills/article-registry/scripts/registry.ts check \
     --source "<出典URL>" --title "<想定タイトル>" --topics "<topic1,topic2,...>"
   ```
   - `duplicate` → スキップ。`review` → 既存記事と実質同じ事案か判断し、続報なら新事実の差分が明確な場合のみ採用。`clear` → 採用。
3. **裏取り**: 採用トピックを一次情報＋複数の独立報道で事実確認。当事者の主張は帰属表現にし、数値は出典と留保を付す。
4. **記事執筆**: 下記「記事フォーマット」と「記事構成」に従い Markdown を `content/articles/<YYYY-MM-DD>-<slug>.md` に作成。
5. **台帳登録**:
   ```bash
   bun run .claude/skills/article-registry/scripts/registry.ts add \
     --slug "<slug>" --title "<タイトル>" --source "<主要出典URL>" \
     --topics "<topic1,topic2,...>" --summary "<要約>" --published "<YYYY-MM-DD>"
   ```
6. これを目安3本繰り返す。
7. **コミット & push**: 記事ファイルと `data/articles.json` を同一コミットに含めて main へ push（push で自動デプロイ）。

## 記事フォーマット（content/articles/<YYYY-MM-DD>-<slug>.md）
フロントマターは **この形式に厳密に** 揃える（パーサが対応するのは `key: value` と `key: [a, b, c]` のみ）。
```markdown
---
slug: kebab-case-unique
title: 中立的で具体的なタイトル
date: 2026-06-14
topics: [ウクライナ, 停戦交渉, 米露, 国連]
sources: [https://www.mofa.go.jp/..., https://www.reuters.com/...]
summary: 記事内容を1〜2文で要約（一覧と meta description に使用）。
---

本文（Markdown）。
```
- ファイル名は日付プレフィックス + slug。`slug` はサイト内で一意。半角英数とハイフン（地域/国名のローマ字など）。
- `topics` は正規化（表記揺れを統一）。地域・国・事案名を使う。`sources` は2件以上、URL のみ。

## 記事構成（良い国際ニュース記事の型）
本文は次の流れを基本にする（見出しは `##`）：

1. **リード（無見出し・冒頭1〜2文）**: 何が・いつ・どこで・誰によって起きたか（5W1H）を端的に。
2. **## 何があったか**: 事実の経緯を時系列で。各当事者の発表は「〜と発表した」「〜と主張した」と帰属。
3. **## 背景・経緯**: 対立や交渉の歴史的文脈、これまでの動き。事実に基づく説明に留める。
4. **## 各国・国際社会の反応**: 当事国双方・関係国・国際機関の立場や声明を併記（中立）。
5. **## 日本への影響・関わり**（該当する場合）: 日本の外交姿勢、邦人・経済・安全保障への影響など。
6. **## 今後の見通し**（任意）: 確定事項と未定事項を区別。断定しない。

末尾の「出典」セクションは frontmatter の `sources` から自動描画されるため本文には不要。

## 注意・落とし穴
- 台帳（`data/articles.json`）を更新せずに記事だけコミットしない（重複検出が壊れる）。
- 紛争当事国の一方的発表を事実として断定しない。死傷者数・戦況・領土支配などは特に帰属と留保を徹底。
- 速報性より正確性。確認できてから扱う。
- 同一事案の続報は「新事実の差分」が明確なときのみ。`summary` に差分を書く。
- 特定の国・勢力・指導者を推奨・非難しない。価値判断を含む表現を避ける。
