# news-collection（政治ニュースの調査・記事化）

## 概要
`news.basaapp.com` は日本の政治ニュースを **毎日・事実ベース・出典付き** で届けるサイト。routine が毎日実行し、信頼できる情報源を調査して **1回あたり3本程度**（上限は設けないが質を優先）の記事を追加する。記事は Markdown で `content/articles/` に置き、main へ push すると自動デプロイされる。

## 編集方針（政治コンテンツの必須ルール）
政治は中立性と正確性が最重要。次を必ず守る：

1. **事実と意見を分離**: 確認できた事実を中心に書く。分析・見通しは「解説」「背景」など明示した区画に限定し、断定しない。
2. **中立・両論併記**: 賛否や与野党で主張が割れる論点は、一方に偏らず複数の立場を併記する。特定政党・人物を擁護/攻撃しない。
3. **出典を必ず明記**: 各記事に最低1件、できれば複数の出典 URL を載せる。一次情報（官公庁・国会・公式発表）を優先し、報道で補う。
4. **憶測・未確認情報を出さない**: 裏取りできない内容、SNS の伝聞、扇情的な見出しは扱わない。確定していない事柄は「〜と報じられている」「〜としている」と帰属を明示。
5. **引用は最小限**: 他サイトの文章をコピーせず、自分の言葉で要約する。引用は短く、出典を示す。
6. **中立的なタイトル**: 煽り・断定・レッテルを避け、何が起きたかを淡々と示す。
7. **出典URLは実在を必ず確認**: `sources` に載せる URL は WebFetch 等で実際に解決し、記事内容と一致することを確認してから書く。記憶や推測で URL を組み立てない（存在しない URL を載せない）。ドメインは正規のもの（例: NHK は `www3.nhk.or.jp`）。

## 情報源（ホワイトリスト軸 + Web 検索補完）
**一次情報を優先**し、報道で肉付けする。下記は初期ホワイトリスト（運用しながら編集してよい）。

一次情報:
- 首相官邸 https://www.kantei.go.jp/
- 各府省庁の公式発表（例: 総務省・財務省・外務省 等の報道発表ページ）
- 国会（衆議院 https://www.shugiin.go.jp/ ・参議院 https://www.sangiin.go.jp/）
- 各政党の公式サイト（政策・声明の一次ソースとして、賛否のバランスを取る）

報道（事実確認・複数ソース照合用。特定社に偏らない）:
- NHK 政治 https://www3.nhk.or.jp/news/
- 主要紙（朝日・読売・毎日・日経・産経 等）を横断し、論調の偏りを相殺する

補完:
- 上記で拾えない最新トピックは **WebSearch** で探し、必ず一次情報か複数報道で裏を取ってから記事化する。

> 単一ソースだけの記事は避け、可能な限り2つ以上で事実確認する。

## ワークフロー（routine 実行時 / 記事1本ごと）

1. **トピック収集**: ホワイトリストの一次情報・報道と WebSearch で、当日の主要な政治トピックを洗い出す（候補を多めに）。
2. **重複チェック（article-registry スキルを使用）**: 各候補について
   ```bash
   bun run .claude/skills/article-registry/scripts/registry.ts check \
     --source "<出典URL>" --title "<想定タイトル>" --topics "<topic1,topic2,...>"
   ```
   - `duplicate` → スキップ。`review` → 既存記事と実質同じテーマか判断し、続報なら新事実の差分が明確な場合のみ採用。`clear` → 採用。
3. **裏取り**: 採用トピックを一次情報＋複数報道で事実確認。確定できない点は帰属表現にする。
4. **記事執筆**: 下記「記事フォーマット」と「記事構成」に従って Markdown を `content/articles/<YYYY-MM-DD>-<slug>.md` に作成。
5. **台帳登録**:
   ```bash
   bun run .claude/skills/article-registry/scripts/registry.ts add \
     --slug "<slug>" --title "<タイトル>" --source "<主要出典URL>" \
     --topics "<topic1,topic2,...>" --summary "<要約>" --published "<YYYY-MM-DD>"
   ```
6. これを目安3本繰り返す。
7. **コミット & push**: 記事ファイルと `data/articles.json` を同一コミットに含めて main へ push（push で自動デプロイ）。

## 記事フォーマット（content/articles/<YYYY-MM-DD>-<slug>.md）
フロントマターは **この形式に厳密に** 揃える（パーサが対応するのは `key: value` と `key: [a, b, c]` のみ）。
```markdown
---
slug: kebab-case-unique
title: 中立的で具体的なタイトル
date: 2026-06-14
topics: [国会, 予算, 与野党]
sources: [https://www.kantei.go.jp/..., https://www3.nhk.or.jp/...]
summary: 記事内容を1〜2文で要約（一覧と meta description に使用）。
---

本文（Markdown）。
```
- ファイル名は日付プレフィックス + slug。`slug` はサイト内で一意。
- `topics` は正規化（表記揺れを統一）。`sources` は1件以上、URL のみ。

## 記事構成（良い政治記事の型）
本文は次の流れを基本にする（見出しは `##`）：

1. **リード（無見出し・冒頭1〜2文）**: 何が・いつ・誰によって起きたか（5W1H）を端的に。
2. **## 何があったか**: 事実の経緯を時系列で。発言は「〜と述べた」と帰属。
3. **## 背景・文脈**: なぜ重要か、これまでの経緯。事実に基づく説明に留める。
4. **## 各党・各方面の反応**: 賛否や与野党の立場を併記（中立）。
5. **## 今後の見通し**（任意）: 確定事項と未定事項を区別。断定しない。

末尾の「出典」セクションは frontmatter の `sources` から自動描画されるため本文には不要。

## 注意・落とし穴
- 台帳（`data/articles.json`）を更新せずに記事だけコミットしない（重複検出が壊れる）。
- 1次情報が取れない速報は無理に出さず、確認できてから扱う。
- 同一テーマの続報は「新事実の差分」が明確なときのみ。`summary` に差分を書く。
- 特定の政治的立場を推奨・批判しない。価値判断を含む表現を避ける。
